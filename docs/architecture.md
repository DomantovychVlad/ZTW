# Архітектура ZortilWatch

Затверджений архітектурний фундамент продукту: шість несучих рішень і конкретний
технічний стек, обраний на їхній основі за результатами технічного дослідження.
Продуктова специфікація — [../PRD.md](../PRD.md); дорожня карта — [../TASKS.md](../TASKS.md);
ліцензійна політика — [licensing.md](licensing.md).

## 1. Форма системи та потік довіри

Два клієнти — **пульт** (керує) і **керований** (його екран показують). Між ними:

- **rendezvous** (сервер з'єднань) лише зводить пристрої за ID і допомагає із сигналінгом — **вмісту сесії не бачить**;
- з'єднання йде **напряму (P2P)**; **coturn (TURN)** вмикається лише як запас і теж бачить тільки зашифровані байти;
- наскрізне шифрування прив'язане до **пароля підключення** (PAKE), тож навіть зламаний сервер не може прочитати чи підмінити сесію.

| Каталог | Роль |
|---------|------|
| `core/` | Rust-ядро: захоплення, кодування, ввід, мережа (WebRTC/NAT), крипто. |
| `ui/` | Спільний React-інтерфейс для десктопа (Tauri) і веб-клієнта. |
| `desktop/` | Оболонка Tauri для Windows. |
| `server/` | rendezvous + coturn relay + акаунти/адресна книга. |
| `docs/` | Архітектура, рішення, правила. |

## 3. Технічний стек за поверхнями

### core/ — Rust (усі ліцензії пермісивні)

| Завдання | Вибір | Ліцензія | Примітка |
|----------|-------|----------|----------|
| WebRTC-транспорт | **str0m** (sans-IO) | MIT/Apache | Стабільний для 24/7-агента; не має вбудованого TURN-клієнта. |
| TURN-клієнт (relay) | `rtc-turn` або власний RFC 8656 | MIT/Apache | Інтеграцію coturn пишемо самі (обмежений обсяг). |
| E2E-крипто (B1) | **spake2** (балансований PAKE) + HKDF + HMAC-прив'язка DTLS-відбитків | MIT/Apache | Альтернатива `pake-cpace` (ISC). Незалежний крипто-аудит до релізу. |
| Захоплення екрана | **DXGI Desktop Duplication** через `windows` (windows-rs) | MIT/Apache | Екран входу/UAC — лише з-під **SYSTEM-агента** (див. ризик №5). |
| Кодування відео | **Media Foundation HW MFT** (H.264 baseline; AV1 де є) | MIT/Apache (bindings) | Zero-copy з DDA. Фолбек: openh264 (бінарник Cisco) або `rav1e` (AV1, без роялті). |
| Декодування (пульт) | — робить браузер/WebView2 WebRTC | — | **Не в Rust** — один медіашлях із веб-клієнтом. |
| Windows-служба | `windows-service` | MIT/Apache | Супервізор Session-0 + сесійний агент. |

> **Заборонено** (порушують F1): `dxgcap`/`dxgcap2`/`captrs` — **AGPL**; будь-які обгортки FFmpeg — **LGPL/GPL**.

### ui/ — TypeScript + React

- Один React-код за **платформним швом** `src/platform/`: десктоп → Tauri IPC до `core`; веб → браузерний WebRTC + WASM-крипто.
- Крипто-ядро з `core` компілюється у **WASM**: `wasm-pack` (target `bundler`) + `vite-plugin-wasm` + `vite-plugin-top-level-await`.
- Дві цілі збірки (Tauri + браузер) з одного коду; `ESLint no-restricted-imports`, щоб платформні імпорти не протікали.

### desktop/ — Tauri v2

- Tauri/wry/tao — MIT/Apache. WebView2 уже є у Windows 10 1803+/11.
- Бандли: **NSIS-інсталятор** (типово) + офлайн-варіант; MSI опційно (лише на Windows).
- ⚠️ Портативний єдиний `.exe` із вшитим WebView2 **неможливий** → рішення на Етапі 11 (ризик №1).

### server/

- **rendezvous** (сигналінг + реєстр онлайн-пристроїв) + **coturn** (TURN, BSD) + **акаунти/адресна книга**.
- Основне розгортання: **Oracle Cloud Always Free, ARM Ampere A1** (не micro — тягне relay-сесії).

## 4. Несучі ризики та відкладені рішення

1. **Портативна версія (Етап 11).** WebView2 не вшивається в один exe. «Портативність» = standalone exe/zip, що покладається на наявний у системі WebView2 (для сучасної Windows прийнятно). Продуктове рішення — на Етапі 11.
2. **Формат одноразового пароля (Етап 3).** 6 цифр ≈ 20 біт — слабко. Рекомендація: 8–9 алфанумерик (~40 біт) + обмеження спроб + ротація + rate-limit на сервері. PAKE дає лише 1 онлайн-спробу, тож обмеження спроб є **обов'язковим**, а не опцією.
3. **Патентні роялті H.264 (юридичне).** Ліцензії *коду* чисті, але H.264 (MPEG-LA/Via-LA) і openh264, зібраний із джерел, несуть патентні роялті для **продаваного** продукту. Чистий від роялті шлях — VP9/AV1 (`rav1e`). Перетинається з C1; потрібен погляд юриста.
4. **str0m без TURN-клієнта.** Обмежений обсяг роботи для relay-фолбеку (E1).
5. **Захоплення секретного робочого стола (Етап 4).** SYSTEM-токен + інжекція в активну консольну сесію + «слідування» за робочим столом і перезапуск агента на кожному UAC/блокуванні — **найбільший інженерний ризик MVP**.
6. **Підпис коду (Етап 11).** Засіб віддаленого керування без OV/EV-сертифіката триґерить SmartScreen/антивіруси.
